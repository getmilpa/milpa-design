Module contracts

Every Milpa module ships with a contract: a machine-readable manifest that declares what the module provides, what it consumes and which events it emits. The kernel never introspects your code — it reads the contract, wires the container and subscribes the listeners before a single line of module code runs.

What is a module contract?

A contract is the public face of a module. It has exactly three clauses: provides lists the services the module binds into the container, consumes names the capabilities it needs from the rest of the field, and emits enumerates every event it may dispatch. Anything not declared is private by definition.

Contracts are also how agents work with a Milpa codebase: an AI pair can read mail.contract.json and know the whole surface of a module without opening its source. That is the contract-first rule — if it is not in the contract, it does not exist.

✦

Added in v0.2.0

Contract linting. coa contract --lint validates every contract against the JSON schema before a harvest — upgrade to v0.2.0 to run it in CI.

Anatomy of a contract

You declare the contract in code — one contract() method per module class — and coa compiles it to JSON at harvest time. The highlighted line is where the container learns about Mailer: consumers depend on the interface, never on the module.

src/Modules/Mail/MailModule.php php 
1<?php23declare(strict_types=1);45namespace App\Modules\Mail;67use Milpa\Contract\Contract;8use Milpa\Module\Module;910final class MailModule extends Module11{12    public function contract(): Contract13    {14        return Contract::define('mail')15            ->provides(Mailer::class) // the kernel binds this for you16            ->consumes('queue', 'config')17            ->emits('mail.sent', 'mail.failed');18    }19}

Declaring a contract

The same contract, three ways to look at it: the fluent PHP builder you write, the CLI view coa prints, and the compiled JSON artifact the kernel actually reads.

MailModule.php php 
Contract::define('mail')    ->provides(Mailer::class)    ->consumes('queue', 'config')    ->emits('mail.sent', 'mail.failed');
$coa contract mail --show
name mail
provides App\Modules\Mail\Mailer
consumes queue, config
emits mail.sent, mail.failed
mail.contract.json json 
{  "name": "mail",  "provides": ["App\\Modules\\Mail\\Mailer"],  "consumes": ["queue", "config"],  "emits": ["mail.sent", "mail.failed"]}

Scaffolding with coa

You rarely write a module from scratch — you plant a seed. coa siembra scaffolds the module class, its contract and a test skeleton in one pass:

$coa siembra mail-module
✔ module class src/Modules/Mail/MailModule.php
✔ contract src/Modules/Mail/mail.contract.json
✔ events src/Modules/Mail/Events (2 files)
✔ test skeleton src/Modules/Mail/Tests/MailModuleTest.php
Done: mail-module planted — 5 files in 212 ms.
⚠

Never edit a compiled contract by hand

mail.contract.json is regenerated from contract() on every harvest. The drift gate fails the build when code and JSON diverge — change the method, not the artifact.

Validating contracts

Contracts are validated twice: at boot in development, and at harvest in CI. When you need it programmatically — custom tooling, editor plugins, agents — go through Contract::validate().

Contract::validate()

Since v0.1.0 
public function validate(ModuleManifest $manifest, bool $strict = true, ?string $schema = null): ValidationReport

Checks a module manifest against the contract schema and the wiring rules of the current field: every consumed capability must be provided by some module, and every emitted event must be namespaced under the module name.

Parameters

Parameters of Contract::validate()
NameTypeDescription
$manifest ModuleManifest The manifest to check, as produced by the module's contract() method.
$strict bool When true, warnings fail the report as well as errors. Defaults to true.
$schema deprecated string|null Path to a custom JSON schema used instead of the bundled one.

Deprecated: $schema is ignored when a Schema instance is bound in the container — bind one instead. The parameter is removed in v0.3.0.

Returns

A ValidationReport with one entry per contract rule; passed() is true when no errors — and, in strict mode, no warnings — were recorded.

Throws

SchemaNotFoundException when the schema path does not resolve to a readable file.

A typical workflow

From bare soil to a registered module, the loop is always the same four moves:

  1. Scaffold the module

    coa siembra mail-module plants the seed: module class, compiled contract and test skeleton.

  2. Declare the contract

    Fill in provides, consumes and emits in the module's contract() method. Keep the surface minimal — private things stay out.

  3. Run the gate

    coa contract --lint checks the contract against the schema; the drift gate compares it with your code and your tests exercise the declared surface.

  4. Harvest

    coa cosecha builds, validates and registers the module with the kernel. The field is one module richer.

Changelog

All notable changes to Milpa. The project follows semantic versioning; breaking changes are called out per item.

v0.2.0

June 26, 2026 latest

Added

  • Contract linting: coa contract --lint validates every contract against the JSON schema before a harvest.
  • Contract::emits() accepts wildcard event names (mail.*) for modules that fan out.

Changed

  • The kernel no longer boots modules without a contract — run coa contract --sync once to generate contracts for legacy modules.
  • ValidationReport is now immutable; mutators return a new instance.

Fixed

  • coa siembra no longer overwrites an existing module directory without --force.
  • Event names are normalized to lowercase before dispatch, matching the contract schema.

v0.1.0

March 3, 2026

Added

  • First public release: the kernel, the module loader and the coa CLI.
  • Contract-first module API — provides, consumes, emits — with Contract::validate().
