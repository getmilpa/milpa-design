Growing an ecommerce plugin from three contracts

bc Bruno Cadenas · Milpa core team Jul 1, 2026

Every ecommerce tutorial starts with a cart screenshot. This one starts with three JSON files. By the end of it we will have a working shop plugin — catalog, cart, checkout — and the only architecture diagram you will need is the one the kernel draws for itself.

Three contracts, no scaffolding

A shop is not one module; it is a small field. We split it into three: catalog (products and prices), cart (a buyer's intent), and checkout (the moment intent becomes an order). Each ships a module contract — a manifest that declares what it provides, what it consumes and which events it emits. Nothing else about the module is public. Not the ORM, not the folder layout, not the helper you regret writing.

That split is the whole trick. Catalog knows nothing about carts. Cart knows nothing about orders. Checkout consumes both and still never imports either — it names capabilities, and the kernel resolves who grows them.

Declaring the checkout surface

Here is the checkout module in full. The highlighted line is its entire public API — one interface. Everything below it is either a declared dependency or a declared event:

src/Modules/Checkout/CheckoutModule.php php 
1<?php23final class CheckoutModule extends Module4{5    public function contract(): Contract6    {7        return Contract::define('checkout')8            ->provides(Checkout::class) // the only public door9            ->consumes('catalog', 'cart', 'payments')10            ->emits('order.placed', 'order.refunded');11    }1213    public function place(Cart $cart): Order14    {15        $order = $this->orders->from($cart); // prices freeze here16        $this->events->emit('order.placed', $order, attempts: 1);17        return $order;18    }19}

Notice what is missing: no binding code, no listener registration, no boot() ceremony. The contract compiles to checkout.contract.json at harvest time, and the drift gate fails the build if the JSON and the method ever disagree.

Letting events do the wiring

Once order.placed is a declared, namespaced event, other modules subscribe to it through their own contracts — and checkout never learns their names:

  • The mail module listens and sends the receipt.
  • The stock module listens and decrements inventory.
  • An analytics plugin we installed a month later listens too — zero changes to checkout.

This is the difference between an event bus and event spaghetti: every emitter is declared, every event is namespaced under its module, and the kernel can print the whole conversation before a single request is served.

We deleted our service locator the week we adopted contracts. The plugin never asks around for what it needs — it declares it, and the field provides.

rv Rufina Vázquez CTO, Kiosko — running the shop plugin in production

What the kernel sees

At boot the kernel reads three manifests — never our code — and wires the container and the listeners from them. The whole plugin, as the machine understands it, fits in one picture:

Harvest day

Shipping is one command. coa cosecha lints every contract against the schema, runs the drift gate, exercises the declared surface with the generated test skeletons, and registers the modules with the kernel. If checkout emits an event it never declared, the harvest fails — in CI, not in production, and never on a Friday night.

The shop plugin is 412 lines of PHP. The three contracts are 31 lines of JSON. Those 31 lines are the ones an AI pair, a new teammate, or the kernel itself actually reads first. Start there, and the rest grows. If you want to plant your own, Milpa 0.2.0 ships everything used in this post.
